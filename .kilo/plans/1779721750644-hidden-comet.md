## 1. PERFORMANCE (49 ‚Üí 90+)

### 1A. Delete Unused `FlickeringGrid` Component (Critical ‚Äî saves ~3.2s JS)
- **File:** `src/components/ui/flickering-grid.tsx`
- The Lighthouse report shows this component's bundle (`flickering-grid.CelZy1Gn.js`) consuming **3,198ms** of JS execution time. It is not imported anywhere in the codebase.
- **Action:** Delete `src/components/ui/flickering-grid.tsx`.

### 1B. Convert Static React Components to Astro (Critical ‚Äî reduces hydration cost)
The homepage loads **5 React islands** (`client:visible`), each adding JS to parse, compile, and hydrate. Three of them have zero interactivity and can be pure Astro components:

1. **`TestimonialSection.tsx`** ‚Äî Pure display, no state/event handlers. Convert to Astro. Replace `lucide-react` Star icons with inline SVGs or `astro-icon`.
2. **`ServicesGrid.tsx`** ‚Äî Pure display, no state/event handlers. Convert to Astro. Pass icon SVGs as slots or inline them.
3. **`HeroWithVideoContent.tsx`** ‚Äî The gradient text styling doesn't need React. Convert to Astro.

This eliminates 3 of 5 client-side hydration points, dramatically reducing TBT and boot-up time.

### 1C. Optimize Wistia Player Loading (High ‚Äî saves ~180ms connection + reduces JS)
- **File:** `src/components/HeadSEO.astro` (or `src/components/WistiaPlayer.astro`)
- **Issue:** No `preconnect` to Wistia domains; 3rd-party scripts total ~320KB (50%+ unused).
- **Actions:**
  1. Add to `<head>`:
     ```html
     <link rel="preconnect" href="https://fast.wistia.com" />
     <link rel="preconnect" href="https://fast.wistia.net" crossorigin />
     <link rel="dns-prefetch" href="https://fast.wistia.com" />
     ```
  2. Consider lazy-loading the Wistia `<script>` with `loading="lazy"` or deferring it until the player is near the viewport (it's in the hero, so preconnect + async is the right balance).
  3. The current `async` attribute on the Wistia script is good ‚Äî keep it.

### 1D. Add Explicit Width/Height to All Images (High ‚Äî fixes CLS)
- **Current CLS:** 0.053 (score: 0.98 ‚Äî borderline)
- **Carousel images** (`FeaturedItemsCarousel.tsx:131`) ‚Äî Missing `width` and `height` attributes. Add them from the known aspect ratio (3:2) and container size.
- **Client logo images** (`ClientLogos.tsx:211`) ‚Äî Missing `width`/`height` on SVG `<img>` tags. Add explicit dimensions (e.g., `width="120" height="48"`).
- **Testimonial avatars** ‚Äî Currently using broken `source.unsplash.com` placeholder URLs (see 1G).

### 1E. Lazy-Load Offscreen Images (High)
- **Issue:** Client logo carousel images and the hidden dark-mode logo in the header load eagerly.
- **Header logos** (`Header.astro:20`) ‚Äî The `dark:inline-block` logo has zero bounding rect (hidden), but still loads. Add `loading="lazy"` to the hidden logo image.
- **Carousel images** already have `loading="lazy"` ‚Äî good.
- **ClientLogos** images need `loading="lazy"` added.

### 1F. Reduce Total Byte Weight (Medium)
- **Current:** 10,737 KiB total.
- **Carousel images** use `-w1200.webp` even for thumbnails in a carousel card. Consider adding `-w600.webp` or `-w800.webp` variants to `targetWidths` in `scripts/optimize-images.mjs` and using smaller sources for carousel cards.
- **OG image** (`public/og-image.jpg`) ‚Äî 369KB. Convert to WebP and compress to ~50KB. Update `HeadSEO.astro` to use `.webp` for OG (note: some social platforms need JPEG fallback).

### 1G. Replace Broken Testimonial Avatar URLs (Medium)
- **File:** `src/components/blocks/TestimonialSection.tsx`
- **Issue:** `https://source.unsplash.com/random/...` returns 403 errors. These broken images add failed network requests.
- **Action:** Remove the `image` prop entirely or replace with initials-only avatars using `AvatarFallback`.

### 1H. Add `fetchpriority="high"` to LCP Image (Medium)
- The hero section background is the likely LCP candidate (or the Wistia thumbnail). Add `fetchpriority="high"` to the first visible logo or primary hero content.

### 1I. Reading Progress Bar ‚Äî Conditional Rendering (Low)
- **File:** `src/components/ReadingProgress.astro` + `src/layouts/BaseLayout.astro`
- **Issue:** The reading progress bar and its scroll listeners are loaded on every page, but only useful on blog posts.
- **Action:** Pass a prop (e.g., `showReadingProgress`) to `BaseLayout` and only render the component on blog post pages.

## 2. ACCESSIBILITY (96 ‚Üí 100)

### 2A. Fix Color Contrast Failures (Critical)
The Lighthouse report flags **insufficient contrast** on these elements:

1. **Hero Badge** (`HeroWithVideoContent.tsx:50-54`) ‚Äî The `<Badge>` with `rounded-full` likely uses `secondary` variant colors. Check the badge background/text contrast ratio. Use a darker text color or adjust the background.
2. **Primary CTA Button** (`HeroWithVideoContent.tsx:71`) ‚Äî The primary button uses sky blue (`202 98% 39%` ‚Üí `hsl(202, 98%, 39%)` ‚âà `#0284c7`) background with white text. This should pass WCAG AA (contrast ratio ~4.6:1), but verify the actual rendered colors match. The issue may be with hover/focus states.
3. **Blog "Read More" Links** (`BlogPostPreview.astro:67`) ‚Äî `buttonVariants({ variant: "link" })` likely renders with a color that fails contrast against the card background.
4. **Footer low-contrast text** (`Footer.astro:29`) ‚Äî `text-muted-foreground/50` (50% opacity) is extremely low contrast. Change to at least `text-muted-foreground/80` or full `text-muted-foreground`.

**Action:** Audit and fix all contrast ratios. Target 4.5:1 for normal text, 3:1 for large text (WCAG AA).

### 2B. Add `aria-label` to Header Navigation (Low)
- **File:** `src/components/Header.astro:23`
- **Action:** Add `aria-label="Main navigation"` to the `<nav>` element.

### 2C. Add `aria-live` to Contact Form Status (Low)
- **File:** `src/pages/contact.astro:99`
- **Issue:** The `#form-status` div doesn't announce changes to screen readers.
- **Action:** Add `aria-live="polite"` to the form status div.

### 2D. Fix FAQ `dangerouslySetInnerHTML` (Low ‚Äî security/accessibility)
- **File:** `src/components/blocks/FaqSectionWithCategories.tsx:107`
- **Issue:** Uses `dangerouslySetInnerHTML` for FAQ answers (to support links). While the content is static/controlled, it's better to use React JSX for rendering. Convert FAQ answers to structured data or use a markdown renderer.
